# 概要设计说明书

## 1. 总体方案

本轮继续基于现有前后端分离架构进行增量实现，不改动项目总体技术栈。

## 2. 真实采集流程设计

1. 新增外部 API Client，V1 负责调用 Gutendex，并接入 Open Library 作为补充来源。
2. 同步服务接收后台同步请求参数，如关键词、语言、主题、数量等。
3. 采集结果转换为内部 `Book` 领域对象后入库。
4. 去重依据继续使用 `source + sourceBookId`。
5. 同步日志需记录请求参数、成功数、失败数、状态和错误信息。
6. 测试环境允许保留样例模式，避免集成测试依赖外部网络。
7. 分类映射继续复用 `BookCategoryService`，通过标题、简介、主题标签做关键词归类。
8. 同步编排层先执行 Gutendex，再根据剩余数量调用 Open Library 补足，并对缺失封面/简介的主源书籍做字段补全。

## 3. 17K 榜单接入设计

1. 新增 `17KRankingSyncService`，负责采集男生作品点击榜和女生作品点击榜。
2. 采集流程拆为榜单页采集和详情页采集两步：
   - 榜单页获取作品链接、榜单类型、排名、点击值
   - 详情页补充书名、作者、分类、简介、封面、状态、原站阅读链接
3. 数据入库时将 `source_channel` 标记为 `17k`，`read_mode` 标记为 `EXTERNAL`。
4. 阅读接口对 `EXTERNAL` 类型返回原站阅读地址，不再返回站内正文地址。
5. 同步日志中记录榜单类型、采集条数、失败条数和来源渠道。

## 4. 书架功能设计

1. 新增 `UserBookshelfEntity`、Mapper、Service、Controller。
2. V1 采用默认用户ID方案，先不引入登录态。
3. 提供书架列表、加入书架、更新进度接口。
4. 前端新增书架页面，并从首页、详情页、阅读页形成联动。

## 5. 后台日志增强设计

1. 扩展日志查询条件，如来源、状态、触发方式。
2. 前端后台页新增日志筛选区和更完整的列展示。
3. 同步任务执行时补充 `requestParams` 和 `errorMessage` 记录。
4. 后台同步区新增关键字、语言、主题、数量输入，以驱动真实采集请求。

## 6. 技术可行方案

推荐采用以下方案：

1. 保留现有 Gutendex + Open Library 站内阅读链路不变。
2. 新增 17K 榜单来源，采集榜单和详情元数据。
3. 17K 来源书籍统一按 `EXTERNAL` 阅读模式处理，在前端跳转原站阅读。
4. 如后续获得合作授权，再扩展 17K 来源为站内阅读模式。
