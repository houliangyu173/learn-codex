# 需求规格说明书

## 1. 需求目标

本轮围绕 BookHub V1 的核心业务增强，补齐真实采集、书架和后台日志三项能力。

## 2. 需求范围

### 2.1 真实采集流程

1. 后台同步不再仅写入样例数据。
2. 系统应调用 Gutendex API 获取真实书籍元数据。
3. V1 以 Gutendex 作为主采集来源，Open Library 作为补充采集与元数据补全来源。
4. 同步结果应继续写入 `book` 和 `book_sync_log`。
5. 当外部接口失败时，系统应给出明确日志和结果说明。
6. 测试环境应允许关闭真实外网采集，改走可控 mock 结果。
7. 当 Gutendex 返回数量不足时，系统应自动尝试使用 Open Library 补足。
8. 当 Gutendex 书籍缺少封面或简介时，系统应允许用 Open Library 补全字段。

### 2.4 17K 榜单接入

1. 系统应支持采集 `www.17k.com` 男生作品点击榜和女生作品点击榜。
2. 系统应支持采集榜单作品的详情元数据，包括书名、作者、分类、简介、封面、榜单类型、排名和详情页链接。
3. 17K 来源作品在 V1 中应按站外阅读来源处理，不在系统内分发正文。
4. 详情页和阅读页应支持展示“前往原站阅读”入口。
5. 后台同步应支持区分男榜、女榜及来源渠道。

### 2.2 书架功能

1. 用户可将书籍加入书架。
2. 用户可查看书架列表。
3. 用户可更新阅读进度。
4. V1 无用户体系时，允许以默认用户ID方式先完成闭环。

### 2.3 后台日志增强

1. 日志查询支持分页。
2. 日志应补充来源、状态、触发方式、成功数、失败数、消息等字段。
3. 后台页面应支持查看最近日志，并支持按来源、状态、触发方式筛选。
4. 日志应展示请求参数和错误信息，便于排查。

## 3. 验收标准

1. 触发同步后，数据库中新增真实来源书籍数据或记录明确失败日志。
2. 同步结果返回主来源、补充来源、补充数量、成功数、失败数和执行消息。
3. 可通过接口或页面完成加入书架、查看书架、更新进度。
4. 进入阅读页后，系统可自动记录基础阅读进度。
5. 后台日志展示信息更完整，并支持基础筛选。
6. 同步日志消息中可体现 Open Library 是否参与补充采集。
7. 17K 来源作品可展示榜单信息并跳转原站阅读。
